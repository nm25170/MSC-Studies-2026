Unit 5 — Reflection: SQL vs. NoSQL Storage Models
London COVID-19 Dataset
MSc Artificial Intelligence | University of Essex
================================================

REFLECTION (~200 words)

The London COVID-19 dataset comprises structured, tabular data with clearly
defined columns (date, area, cases, deaths, admissions) and a fixed schema.
This structure is well-suited to a relational SQL model. SQL databases enforce
schema integrity and support complex queries such as aggregations, joins, and
time-series filtering — operations central to epidemiological analysis
(Date, 2004). Tools such as PostgreSQL or MySQL would allow efficient execution
of queries across the 110 weekly records, maintaining data consistency through
ACID (Atomicity, Consistency, Isolation, Durability) compliance
(Stonebraker, 2010).

However, if the dataset were expanded to integrate real-time feeds, unstructured
clinical notes, or geospatial borough-level data, a NoSQL model would offer
advantages. Document-oriented databases such as MongoDB support flexible schemas
and horizontal scalability, accommodating the heterogeneous formats common in
public health surveillance systems (Sadalage and Fowler, 2012). NoSQL databases
prioritise availability and partition tolerance under the CAP theorem, making
them better suited for high-volume ingestion pipelines.

For this use case — a structured, fixed-schema weekly dataset analysed with
Pandas — SQL remains the more appropriate model. NoSQL would be preferred if
the data volume, velocity, or variety increased significantly.
